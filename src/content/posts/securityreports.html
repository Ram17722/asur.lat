Creating Security Reports

Cybersecurity Incident Response

Introduction

Security reports are a critical component of any cybersecurity response strategy. They help technical teams, management, and stakeholders understand the nature of an incident, assess its impact, and make informed decisions for recovery and future protection. This post outlines how to write an effective security report, using a Denial of Service (DoS) attack as a working example.

1. Executive Summary

The executive summary provides a high-level overview of the incident. It should briefly describe what happened, the type of attack, and the overall business impact. This section is intended for upper management or non-technical stakeholders who need quick insight into the incident.

  • Example: A SYN flood DoS attack targeted our public-facing API, causing service degradation for 2 hours. No data breach occurred, and customer data remained secure. The issue was resolved the same day.

2. Technical Details

This section provides the in-depth technical explanation of the incident. It includes how the attack was carried out and which vulnerabilities or systems were exploited. Diagrams or flowcharts can be included if necessary to visualize the attack vector.

  • Type of attack: SYN flood (TCP handshake exploitation)
  • Attack vector: Excessive half-open connections overloaded the server
  • Affected systems: Load balancer, web application gateway, public API endpoints
  • Packet characteristics: Spoofed IPs, consistent TTL, SYN-only flags

3. Incident Timeline

Present a detailed chronological sequence of events to clearly show detection, escalation, and remediation milestones.

  • 08:15 AM: Elevated SYN traffic detected in NetFlow logs
  • 08:30 AM: Services begin to degrade, triggering alerts
  • 08:45 AM: Incident escalated to security team
  • 09:10 AM: Mitigation implemented (rate-limiting via iptables)
  • 10:20 AM: Full service restored and DDoS shield engaged

4. Detection and Response

Explain how the attack was discovered and what actions were taken in response. Mention specific tools and teams involved in identification and mitigation.

  • Detected via IDS alerts and anomaly-based monitoring tools (Snort, Zabbix)
  • Initial mitigation: Blocked malicious IP ranges, rate-limiting on SYN packets
  • Response team: Security Operations Center (SOC), Network Engineering, DevOps
  • Escalation flow: Automated alerts → SOC triage → Engineering involvement

5. Impact Analysis

Detail the operational and potential security impact. Include any measurable metrics and customer-facing effects.

  • Service impact: API latency increased, 12% of requests failed
  • Customer impact: Minor disruption reported by 2 enterprise clients
  • Security impact: No unauthorized access or data compromise
  • Financial/reputation: No SLA breach occurred; no media coverage

6. Mitigation and Recovery

Outline the steps taken to stop the attack and return to normal operations. Distinguish between immediate and long-term measures.

  • Short-term: Rate-limiting and firewall rules (iptables, AWS WAF)
  • Long-term: Deployment of SYN cookies, reconfiguration of load balancer thresholds
  • Additional: Subscription to third-party DDoS protection (Cloudflare Magic Transit)
  • Recovery status: Normal operations resumed; active monitoring in place

7. Lessons Learned

Reflect on what went well and what could be improved in future responses. This section is key for continuous improvement.

  • Effective: Early detection through well-tuned monitoring; coordinated team response
  • Needs improvement: Faster alert escalation to engineering; better documentation of network baseline traffic
  • Communication: Incident notification template lacked some key details for execs

8. Recommendations

Provide actionable advice to prevent or respond to similar incidents in the future. Focus on both people and technology.

  • Enhance network monitoring thresholds and alert tuning
  • Run quarterly incident response simulations including DoS/DDoS scenarios
  • Implement preconfigured emergency firewall rules for rapid activation
  • Review and update incident runbooks for clarity

  • Firewall logs showing high SYN counts from spoofed IPs
  • Traffic graphs illustrating abnormal spikes in TCP SYN
  • Packet capture excerpts demonstrating incomplete handshakes
  • Alerts triggered by monitoring tools with timestamps

Conclusion

Writing clear and structured security reports helps organizations respond to incidents more effectively and prepare for future threats. By using a consistent format and providing both technical and operational perspectives, your report becomes a valuable tool for continuous security improvement. Documenting incidents properly also helps inform policy decisions and drive investment in the right defenses.
